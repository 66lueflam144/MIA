又告诉你不行——因为flag被过滤了。

解释一下就是：

    1. PING 127.0.0.1 (127.0.0.1): 56 data bytes：这是一个命令行输出，表示将对IP地址 127.0.0.1 进行Ping测试。/?ip=：这是一个URL参数，用于接收要测试的目标IP地址。
    1. |\'|\"|\\|\(|\)|\[|\]|\{|\}/", $ip, $match))：这是一个正则表达式，用于检查输入的IP地址是否包含特定的符号（ ', ", \, (, ), [, ], {, }）。然后 echo preg_match("/\&|\/|\?|\*|\<|[\x{00}-\x{20}]|\>|\'|\"|\\|\(|\)|\[|\]|\{|\}/", $ip, $match);：这一行判断输入的IP地址是否包含特殊字符，并输出匹配结果。最后die("fxck your symbol!");：如果前面的正则表达式匹配到特殊字符，将输出错误信息并终止程序运行。

HEXO+Github Pages
2023/10/17

HEXO

HEXO+GITHUB PAGES

因为之前做的那个博客上传出现了问题，所以又搭建一个，其实也可以用docsify的，就是好像css渲染出现问题来着就放弃了。

  • 前提是Node.js你配置好了。
  • 接着就是安装hexo框架，我们通过cnpm（npm国内镜像慢到你绝望甚至连接不上），所以先安装cnpm
npm install -g cnpm --registry=https://registry.npm.taobao.org
  • 安装hexo框架
cnpm install -g hexo-cli
  • 接着建一个仓库，不一定要求是xxx.github.io这种名字
  • clone至本地
  • hexo初始化
hexo init xxx

需要注意的是，xxx是指你在你clone的仓库文件里新的文件夹（不需要你自建因为这个命令就是创建且初始化但是建了也没关系），不是指你仓库名（因为hexo拒绝非空文件夹的初始化）

  • 初始化后hexo s就可以启动本地查看。（可看可不看，这不是重点，一般用于预览）
  • 部署到github：
    • 在初始化过的文件夹里cnpm install --save hexo-deployer-git安装必要的插件
    • 初始化后的文件夹里有一个_config.yml文件，进入对一些数据进行修改（记得里面的都需要在:后空格才能生效）：
      • # Site部分就是修改自己博客的title、博客主人名之类
      • # URL部分下的url修改为你那个仓库的URL，是带.git结尾的
      • 还要添加一个root，填写仓库名，比如/yourname.github.io/。root有无对io仓库可能没什么影响但对非io仓库影响很大——可能css，js的渲染加载不出来。
      • theme部分是在你使用了其他theme后修改，后面可以具体说。
      • # Deployment部分，type: git,repo: https://xxx/xxx.git,branch: gh-pages
  • 搞完上述的配置之后，命令行输入每次更新必输入的三句命令：
    • hexo clean
    • hexo g
    • hexo d（第2和第3可以合并为hexo g -d）
  • hexo连接是一个谜，所以有时候会报错，失败了也不用着急，全随缘，没准什么时候就d成功了。
  • 一些其他命令hexo new post xxx，就会自动创建新文件，目录它会自动告诉你。

PLUS theme

找到一个喜欢的主题xxx，下载解压到博客文件夹的theme文件夹里面，一般theme都会有使用手册，按照里面的要求进行修改就行。
然后再回到根目录下的_config.yml文件里修改theme: xxx。最后三句命令进行更新推送，等待page build好就可以看到了。

  • php中有几个超全局变量，$_POST是其中一个。
    $_POST['shell']意为用POST的方法接收shell变量。
  • eval()函数将字符串作为PHP代码执行
